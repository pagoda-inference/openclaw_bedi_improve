# 元素分析参考文档

本文档提供页面元素交互性分析的方法论和框架，帮助智能体识别可交互元素并判断其安全性。

## 分析目标

元素分析的核心目标是理解：
1. **交互类型** - 元素如何被操作
2. **交互目的** - 操作会产生什么结果
3. **安全性判断** - 操作是只读还是写入（关键）
4. **导航价值** - 操作是否能发现新的页面URL
5. **元素关系** - 元素之间如何协作

## 分析维度

### 1. 交互类型识别

**常见交互类型：**

| 交互类型 | 触发方式 | 典型元素 | 使用场景 |
|---------|---------|---------|---------|
| click | 鼠标点击 | button, a, div | 导航、提交、触发操作 |
| type | 键盘输入 | input, textarea | 表单填写、搜索 |
| select | 下拉选择 | select, dropdown | 选项选择、筛选 |
| scroll | 滚动操作 | div, body | 内容浏览、加载更多 |
| hover | 鼠标悬停 | a, div | 显示菜单、提示信息 |
| drag | 拖拽操作 | div, img | 排序、移动元素 |

**识别方法：**
- 查看HTML标签类型（button, input, a等）
- 检查事件监听器（onclick, onchange等）
- 观察CSS样式（cursor: pointer等）
- 测试实际交互行为

### 2. 只读/写入分类（关键维度）

这是 Deep Browser 中最重要的分析维度，决定元素是否可以安全操作。

**只读类元素（允许操作）：**

| 元素特征 | 交互结果 | 示例 |
|---------|---------|------|
| 导航链接 | 跳转到新页面 | `<a href="/products">` |
| 菜单项 | 显示子菜单或跳转 | 导航菜单项 |
| 展开按钮 | 显示隐藏内容 | 手风琴、折叠面板 |
| 标签切换 | 切换内容区域 | 选项卡 |
| 搜索框+搜索按钮 | 展示搜索结果 | 搜索表单 |
| 筛选控件 | 改变内容展示 | 下拉筛选、复选框 |
| 排序控件 | 改变排列顺序 | 排序按钮 |
| 分页控件 | 加载其他页数据 | 下一页、页码 |
| 滚动触发 | 加载更多内容 | 无限滚动 |

**写入类元素（禁止操作）：**

| 元素特征 | 交互结果 | 示例 |
|---------|---------|------|
| 注册提交 | 创建新账号 | 注册表单提交按钮 |
| 登录提交 | 创建用户会话 | 登录表单提交按钮 |
| 联系表单提交 | 发送消息 | 联系我们提交按钮 |
| 购买/下单 | 产生交易 | 购买按钮、结算按钮 |
| 编辑提交 | 修改已有数据 | 编辑表单保存按钮 |
| 删除操作 | 删除数据 | 删除按钮 |
| 发布/上传 | 写入新数据 | 发布按钮、上传按钮 |
| 评论提交 | 发表评论 | 评论提交按钮 |

**判断方法：**
- 分析元素的文本内容（"提交"、"购买"、"注册" vs "搜索"、"筛选"、"下一页"）
- 查看元素所在的表单类型（搜索表单 vs 注册表单）
- 检查表单的action属性（/search vs /register）
- 理解业务上下文（该操作是读取数据还是写入数据）

**核心判断标准**：
- 如果操作的目的是**获取/展示信息** → 只读 → ✅ 允许
- 如果操作的目的是**创建/修改/删除数据** → 写入 → ❌ 禁止

### 3. 导航价值评估

评估交互元素是否能帮助发现新的页面URL。

**高导航价值：**

| 元素类型 | 发现方式 | 导航价值 |
|---------|---------|---------|
| 导航链接 | 直接点击 | ⭐⭐⭐ 高 |
| 面包屑 | 直接点击 | ⭐⭐⭐ 高 |
| 分页控件 | 直接点击 | ⭐⭐⭐ 高 |
| 搜索结果 | 搜索后点击 | ⭐⭐⭐ 高 |

**中导航价值：**

| 元素类型 | 发现方式 | 导航价值 |
|---------|---------|---------|
| 折叠菜单 | 展开后发现链接 | ⭐⭐ 中 |
| 标签切换 | 切换后发现链接 | ⭐⭐ 中 |
| 下拉菜单 | 展开后发现链接 | ⭐⭐ 中 |
| 滚动加载 | 滚动后发现新条目 | ⭐⭐ 中 |

**低导航价值：**

| 元素类型 | 发现方式 | 导航价值 |
|---------|---------|---------|
| 筛选控件 | 改变展示但不产生新URL | ⭐ 低 |
| 排序控件 | 改变顺序但不产生新URL | ⭐ 低 |
| 悬停效果 | 临时展示 | ⭐ 低 |

### 4. 交互目的推断

**常见目的类型：**

| 目的类型 | 特征 | 安全性 | 示例 |
|---------|------|--------|------|
| 导航 | 跳转到其他页面 | ✅ 只读 | 链接、菜单项 |
| 展示 | 展示/隐藏内容 | ✅ 只读 | 展开/折叠按钮 |
| 搜索 | 获取搜索结果 | ✅ 只读 | 搜索框+按钮 |
| 筛选 | 修改显示内容 | ✅ 只读 | 下拉选择、复选框 |
| 提交 | 创建/修改数据 | ❌ 写入 | 注册、登录、联系表单 |
| 操作 | 执行特定功能 | ⚠️ 需判断 | 购买（写入）、下载（只读） |

**推断方法：**
- 分析元素文本内容（按钮文字、链接文本）
- 查看元素位置（导航栏、表单中、内容区）
- 检查相关元素（表单的提交按钮）
- 理解上下文（搜索框旁边的搜索按钮）
- 检查表单action属性

### 5. 重要性评估

**重要性等级（结合导航价值和安全性）：**

```
高 (high) - 优先处理
├─ 导航入口（链接、菜单）→ 发现新页面
├─ 展示交互（折叠、标签）→ 发现隐藏内容
└─ 搜索/筛选 → 获取更多数据

中 (medium) - 次要处理
├─ 辅助导航（面包屑、页脚链接）
├─ 分页控件 → 获取更多条目
└─ 排序/筛选 → 改变展示方式

低 (low) - 记录但不优先
├─ 装饰性元素
├─ 社交分享按钮
└─ 辅助功能

禁止 (forbidden) - 记录但不执行
├─ 表单提交按钮（注册、登录、联系）
├─ 购买/下单按钮
├─ 编辑/删除按钮
└─ 发布/上传按钮
```

### 6. 元素分组

**常见分组类型：**

| 分组类型 | 特征 | 示例 |
|---------|------|------|
| navigation | 导航相关元素 | 主导航：首页+产品+关于 |
| form-read | 读取类表单 | 搜索表单：搜索框+搜索按钮 |
| form-write | 写入类表单 | 注册表单：用户名+密码+提交 |
| list | 列表项 | 商品列表：多个商品卡片 |
| menu | 菜单项 | 下拉菜单：多个选项 |
| accordion | 折叠面板 | FAQ：多个可展开问题 |

**分组方法：**
- 查找父容器元素（form, nav, ul）
- 识别共同功能（都是搜索相关）
- 分析空间关系（相邻的元素）
- 理解逻辑关系（输入+提交）
- 判断表单分组是读取类还是写入类

## 分析工具

### 浏览器开发工具

**元素检查：**
```javascript
getEventListeners(element)
element.attributes
window.getComputedStyle(element)
```

**安全性判断辅助：**
```javascript
// 检查表单action
form.action
form.method

// 检查链接目标
element.href

// 检查按钮类型
element.type  // "submit" vs "button"

// 检查表单字段
form.elements
```

**交互测试：**
```javascript
element.click()
element.value = 'test'
element.dispatchEvent(new Event('input'))
element.isContentEditable
element.disabled
```

### 选择器定位

**CSS选择器：**
```javascript
document.querySelectorAll('a')           // 导航链接
document.querySelectorAll('button')       // 按钮
document.querySelectorAll('input[type="search"]')  // 搜索框
document.querySelectorAll('form')         // 表单
document.querySelectorAll('[role="button"]')       // 可点击元素
```

**XPath：**
```javascript
//button[contains(text(), '提交')]
//input[@type='text']
//a[contains(@href, 'product')]
//form[contains(@action, 'search')]
```

## 分析流程

### 标准流程

```
1. 页面扫描
   ├─ 获取所有元素
   ├─ 识别元素类型
   └─ 标记可见性

2. 安全性分类（关键步骤）
   ├─ 判断每个元素是只读还是写入
   ├─ 标记只读元素为可操作
   └─ 标记写入元素为禁止操作

3. 导航价值评估
   ├─ 评估每个只读元素的导航价值
   ├─ 高导航价值 → 优先执行
   └─ 低导航价值 → 次要处理

4. 目的推断
   ├─ 分析元素文本
   ├─ 查看上下文
   └─ 理解功能逻辑

5. 分组归类
   ├─ 查找父容器
   ├─ 识别功能组
   ├─ 区分读取类表单和写入类表单
   └─ 记录关系
```

### 特殊情况处理

**隐藏元素：**
- 检查display: none
- 检查visibility: hidden
- 检查opacity: 0
- 标记为不可见

**动态元素：**
- 等待元素出现
- 监听DOM变化
- 使用动态选择器

**禁用元素：**
- 检查disabled属性
- 检查aria-disabled
- 标记为不可交互

**模糊判断：**
- 如果无法确定元素是只读还是写入 → 视为写入类，禁止操作
- 如果表单目的不明确 → 视为写入类，禁止操作
- 宁可漏掉一个导航入口，也不要执行写入操作

## 最佳实践

1. **安全性优先**
   - 每次交互前必须判断安全性
   - 无法确定时视为写入类
   - 只执行明确的只读操作

2. **导航价值优先**
   - 优先处理高导航价值的元素
   - 每个页面的首要任务是发现导航入口
   - 不遗漏任何可能发现新页面的交互

3. **上下文理解**
   - 结合页面类型分析
   - 考虑用户目标
   - 理解业务逻辑

4. **持续更新**
   - 页面变化时重新分析
   - 更新元素列表
   - 调整安全性和导航价值判断

## 常见问题

**Q: 如何判断元素是只读还是写入？**
A: 分析元素文本、表单action、业务上下文。如果操作目的是获取/展示信息则为只读，如果目的是创建/修改数据则为写入。

**Q: 无法确定安全性时怎么办？**
A: 视为写入类，禁止操作。宁可漏掉一个导航入口，也不要执行写入操作。

**Q: 搜索表单和注册表单如何区分？**
A: 搜索表单通常有搜索框和搜索按钮，action包含search/query等关键词；注册表单有用户名/密码/邮箱等字段，action包含register/signup等关键词。

**Q: 如何处理动态加载的元素？**
A: 等待元素加载完成，或使用MutationObserver监听DOM变化，然后对新元素进行安全性判断
