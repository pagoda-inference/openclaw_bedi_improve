# 任务隔离设计

## 一、核心问题

每次调研任务可能涉及不同的主题、数据来源、分析维度和输出要求。如果任务之间没有隔离，会导致：

- 数据混乱：不同任务的数据混在一起
- 记忆污染：一个任务的经验影响另一个任务
- 状态混乱：无法区分当前任务进度
- 难以追溯：无法回溯历史任务

---

## 二、隔离方案：基于文件夹的任务隔离

### 2.1 核心思路

每个任务创建一个独立的文件夹，该任务的所有数据、记忆、输出都存储在这个文件夹中。

**隔离原则**：
- 任务之间完全隔离，互不干扰
- 通过任务 ID 区分不同任务
- 支持任务暂停、恢复、归档

---

## 三、文件夹结构设计

### 3.1 整体结构

```
~/.openclaw/
├── tasks/                      # 任务目录
│   ├── task-20250427-001/      # 任务 1
│   ├── task-20250427-002/      # 任务 2
│   └── ...
│
├── archive/                    # 归档目录
│   ├── task-20250420-001/      # 已归档任务
│   └── ...

│   └── knowledge/              # 知识库
│
└── config/                     # 全局配置
    └── settings.json
```

## 四、核心文件说明

### 4.1 task.json - 任务元信息

```json
{
  "task_id": "task-20250427-001",
  "task_type": "keyword_research",
  "status": "in_progress",
  "user_input": "调研 AI Agent 市场规模和主要玩家",
  "created_at": "2025-04-27T10:00:00Z",
  "updated_at": "2025-04-27T10:30:00Z",
  "started_at": "2025-04-27T10:00:05Z",
  "completed_at": null,
  "progress": {
    "phase": "data_collection",
    "percentage": 45,
    "current_action": "访问第 5 个网站"
  }
}
```

**字段说明**：
- `task_id`：任务唯一标识，格式为 `task-{日期}-{序号}`
- `task_type`：任务类型（keyword_research / site_analysis / mixed）
- `status`：任务状态（pending / in_progress / paused / completed / cancelled）
- `user_input`：用户原始输入
- `progress`：任务进度信息

### 4.2 config.json - 任务配置

```json
{
  "research_depth": "standard",
  "output_formats": ["html", "pdf"],
  "time_limit_minutes": 30,
  "max_pages": 100,
  "target_sites": [],
  "exclude_sites": [],
  "language": "zh-CN",
  "report_template": "industry_overview"
}
```

**字段说明**：
- `research_depth`：调研深度（quick / standard / deep）
- `output_formats`：输出格式列表
- `time_limit_minutes`：时间限制（分钟）
- `max_pages`：最大访问页面数
- `target_sites`：指定网站列表（可选）
- `exclude_sites`：排除网站列表（可选）

## 五、任务状态管理

### 5.1 任务状态流转

```
pending → in_progress → completed
              ↓
           paused → in_progress
              ↓
           cancelled
```

**状态说明**：
- `pending`：任务已创建，等待执行
- `in_progress`：任务正在执行
- `paused`：任务暂停，可恢复
- `completed`：任务完成
- `cancelled`：任务取消

### 5.2 状态文件更新规则

每次状态变更时：
1. 更新 `task.json` 中的 `status` 字段
2. 更新 `updated_at` 时间戳
3. 记录到 `logs/task.log`

## 六、共享资源

### 6.1 共享资源目录

```
shared/
├── patterns/                   # 网站模式库（跨任务共享）
│   ├── layouts/                # 布局模式
│   │   ├── ecommerce-list.json
│   │   └── blog-article.json
│   │
│   ├── sites/                  # 网站模式
│   │   ├── amazon.com.json
│   │   └── github.com.json
│   │
│   └── skills/                 # 操作技能
│       ├── pagination.json
│       └── search.json
│
└── knowledge/                  # 知识库（跨任务共享）
    ├── site-features.json      # 网站特征
    └── best-practices.json     # 最佳实践
```

### 6.2 共享资源访问规则

**读取**：任何任务都可以读取共享资源
**写入**：只有任务成功完成后，才能将新模式写入共享资源
**更新**：更新共享资源需要验证，避免覆盖有效模式

## 七、任务归档

### 7.1 归档条件

- 任务完成超过 7 天
- 用户手动归档
- 存储空间不足时自动归档最旧任务

### 7.2 归档操作

1. 压缩任务文件夹
2. 移动到 `archive/` 目录
3. 保留 `task.json` 元信息用于追溯
4. 清理原始任务文件夹

### 7.3 归档后结构

```
archive/
├── task-20250420-001.zip       # 压缩的任务数据
├── task-20250420-002.zip
└── index.json                  # 归档索引
```

---

## 八、任务恢复

### 8.1 恢复条件

任务处于 `paused` 或 `in_progress` 状态时可以恢复。

### 8.2 恢复流程

1. 读取 `task.json` 获取任务状态
2. 读取 `memory/operations.json` 获取操作历史
3. 读取 `memory/visited.json` 获取已访问 URL
4. 从上次中断点继续执行

---

## 九、总结

| 隔离维度 | 实现方式 | 说明 |
|---------|---------|------|
| 数据隔离 | 独立文件夹 | 每个任务独立的数据目录 |
| 记忆隔离 | 任务级文件 | operations.json、visited.json |
| 配置隔离 | 独立配置文件 | config.json |
| 输出隔离 | 独立输出目录 | output/ |
| 日志隔离 | 独立日志文件 | logs/ |

**核心优势**：
1. 简单直观：基于文件夹，易于理解和调试
2. 完全隔离：任务之间互不干扰
3. 易于追溯：保留完整的任务历史
4. 支持恢复：可从任意点恢复任务
