# OpenClaw 市场调研智能体系统优化方案

## 一、项目背景与目标

### 1.1 项目背景

OpenClaw 是一个智能体系统，具备基础的浏览器操作、网页抓取、LLM 推理等能力。当前在市场调研场景下，存在以下不足：

- 浏览器操作能力较浅，难以深度理解和操作复杂网页
- 缺乏任务隔离机制，多任务并行时容易混乱
- 数据处理能力有限，缺乏系统化的清洗、分析流程
- 报告生成能力薄弱，难以输出专业、美观的报告

### 1.2 优化目标

将 OpenClaw 改造成一个**适合市场调研-报告生成的智能体系统**，核心目标：

1. **深度浏览器操作**：让智能体能够深度理解、操作、观察、记忆网页系统
2. **任务隔离执行**：每个调研任务独立隔离，互不干扰
3. **系统化数据处理**：标准化的数据清理、清洗、分析流程
4. **专业报告生成**：智能选择模板，生成美观的 HTML 报告

### 1.3 核心原则

- **综合改造**：合理使用所有方案，包括核心改造、Skill 扩展、工具集成等
- **渐进式实现**：分阶段实现，先验证核心价值再增强功能
- **学习型系统**：通过记忆机制，让系统越用越聪明
- **成本优化**：借鉴 OpenCLI 思路，一次学习，多次零成本执行

## 二、整体架构设计

### 2.1 改造后的系统架构

OpenClaw 改造后的核心架构分为四层：

**核心层**
- Agent Engine：智能体引擎，负责任务调度和执行
- LLM Provider：大语言模型接口
- Memory System：记忆系统（增强为三层架构）

**能力层**
- Deep Browser：深度浏览器操作能力（新增）
- Web Tools：web_search、web_fetch（保留）
- Data Processor：数据处理能力（新增）
- Report Generator：报告生成能力（新增）

**存储层**
- Task Storage：任务隔离存储（新增）
- Shared Storage：共享资源存储（新增）
- File System：文件系统（保留）

**输出层**
- Gateway：HTTP 服务（保留）
- Canvas：展示层（保留）

### 2.2 核心模块说明

| 模块 | 职责 | 改造类型 |
|------|------|---------|
| **Deep Browser** | 深度浏览器操作：理解、操作、观察、记忆 | 新增 |
| **Task Manager** | 任务创建、隔离、恢复、归档 | 新增 |
| **Memory System** | 三层记忆：会话、模式、知识 | 增强 |
| **Data Processor** | 数据清理、清洗、分析 | 新增 |
| **Report Generator** | 报告模板选择、内容组织、HTML 渲染 | 新增 |

### 2.3 数据流转

```
用户输入
    ↓
任务创建（Task Manager）
    ↓
调研规划（Agent Engine + LLM）
    ↓
数据采集（Deep Browser）
    ↓
数据处理（Data Processor）
    ↓
报告生成（Report Generator）
    ↓
输出交付（Gateway）
```

## 三、Deep Browser 模块设计

### 3.1 核心定位

Deep Browser 是 OpenClaw 系统级别的浏览器能力增强模块，让智能体能够深度理解、操作、观察、记忆网页系统。

### 3.2 四大核心能力

**理解能力（Understand）**
- 页面类型识别：电商列表、博客文章、搜索结果、文档站等
- 页面结构理解：布局、导航、内容区域
- 交互元素识别：可点击、可输入、可滚动的元素
- 数据结构识别：表格、列表、卡片等数据组织方式

**操作能力（Operate）**
- 基础操作：导航、点击、输入、滚动、等待
- 复合操作：翻页、搜索、筛选、展开
- 策略操作：根据目标选择最优操作序列

**观察能力（Observe）**
- 变化检测：页面是否变化、变化了什么
- 效果验证：操作是否成功
- 状态感知：当前处于什么状态
- 异常识别：弹窗、验证码、错误提示

**记忆能力（Remember）**
- 会话记忆：操作历史、已访问 URL
- 模式记忆：网站布局模式、操作技能模板
- 知识记忆：网站特征知识、最佳实践

### 3.3 工作循环

Deep Browser 采用持续循环的工作方式：

**目标 → 理解 → 规划 → 操作 → 观察 → 记忆 → 判断完成**

关键特点：
- 反馈驱动：每次操作后都有观察和验证
- 记忆积累：每次循环都更新记忆
- 动态调整：根据观察结果调整策略

### 3.4 与 OpenCLI 的集成

借鉴 OpenCLI 的核心思想：**编译期智能 vs 运行期智能**

- 首次访问网站：用 LLM 分析，生成模式描述
- 后续访问：直接使用模式，零 LLM 成本
- 频繁访问：生成确定性适配器，100% 稳定执行

集成方式：
1. 直接使用 OpenCLI 的浏览器命令
2. 利用 OpenCLI 的预置适配器
3. 生成新的 OpenCLI 适配器

## 四、任务隔离模块设计

### 4.1 核心思路

每个任务创建一个独立的文件夹，该任务的所有数据、记忆、输出都存储在这个文件夹中。

### 4.2 文件夹结构

```
~/.openclaw/
├── tasks/                      # 任务目录
│   └── task-20250427-001/      # 单个任务
│       ├── task.json           # 任务元信息
│       ├── config.json         # 任务配置
│       ├── plan.json           # 调研计划
│       ├── data/               # 数据目录
│       │   ├── raw/            # 原始数据
│       │   ├── processed/      # 处理后数据
│       │   └── analysis/       # 分析结果
│       ├── memory/             # 任务记忆
│       │   ├── operations.json # 操作历史
│       │   ├── visited.json    # 已访问 URL
│       │   └── decisions.json  # 决策记录
│       ├── output/             # 输出目录
│       │   ├── report.html     # 生成的报告
│       │   └── assets/         # 报告资源
│       └── logs/               # 日志目录
│
├── archive/                    # 归档目录
│
├── shared/                     # 共享资源（跨任务）
│   ├── patterns/               # 网站模式库
│   └── knowledge/              # 知识库
│
└── config/                     # 全局配置
```

### 4.3 任务状态流转

```
pending → in_progress → completed
              ↓
           paused → in_progress
              ↓
           cancelled
```

### 4.4 任务恢复

任务处于 `paused` 或 `in_progress` 状态时可以恢复：

1. 读取 `task.json` 获取任务状态
2. 读取 `memory/operations.json` 获取操作历史
3. 读取 `memory/visited.json` 获取已访问 URL
4. 从上次中断点继续执行

## 五、数据处理模块设计

### 5.1 处理流程

**原始数据 → 数据清理 → 数据清洗 → 数据分析 → 结构化输出**

### 5.2 数据清理

**噪声去除**
- 去除 HTML 标签、CSS 样式、JavaScript 代码
- 去除广告内容、导航文字
- 去除特殊字符、多余空白

**缺失值处理**
- 识别空值、null、占位符
- 选择删除、保留或推断

**重复数据处理**
- 完全重复：合并为单条
- 部分重复：合并互补信息
- 语义重复：语义相似度合并

**格式标准化**
- 时间格式统一
- 货币格式统一
- 数值格式统一
- 名称格式统一

### 5.3 数据清洗

**数据验证**
- 格式验证：日期、数值、枚举值
- 逻辑验证：总量=各部分之和、增长率计算
- 业务验证：符合业务常识

**异常检测**
- 数值异常：极端值、异常增长率
- 逻辑异常：自相矛盾、时间线异常
- 模式异常：异常分布、异常关联

**数据修正**
- 格式修正、内容修正、来源修正

### 5.4 数据分析

**统计分析**
- 描述性统计：计数、求和、平均、中位数
- 分布分析：数据分布、分位数
- 占比分析：各部分占比

**趋势分析**
- 时间序列分析：趋势方向、周期性
- 增长率分析：同比、环比、复合增长
- 预测分析：基于历史预测

**对比分析**
- 横向对比：不同对象、地区、类别
- 纵向对比：不同时期、历史对比
- 差异分析：识别显著差异

**关联分析**
- 相关性分析：相关系数
- 因果关系分析：因果链条
- 聚类分析：相似数据群组

## 六、报告生成模块设计

### 6.1 处理流程

**分析结果 → 模板选择 → 内容组织 → HTML 渲染 → 输出交付**

### 6.2 模板系统

**6 种内置模板**

| 模板 | 适用场景 |
|------|---------|
| 行业概览 | 市场规模、竞争格局、发展趋势 |
| 竞品对比 | 产品对比、优劣势分析 |
| 深度研究 | 背景分析、现状调研、前景展望 |
| 趋势追踪 | 事件追踪、热度变化 |
| 快速简报 | 快速了解、核心要点 |
| 数据报告 | 数据密集型分析 |

**模板选择**

用户未指定模板时，使用 LLM 智能选择：

```
用户需求：{user_input}
数据摘要：{data_summary}

可用模板：A.行业概览 B.竞品对比 C.深度研究 D.趋势追踪 E.快速简报 F.数据报告

请选择最合适的模板，返回选项字母。
```

### 6.3 内容组织

**自动映射**
- 将分析结果映射到模板章节
- 智能填充必填章节
- 缺失内容标注"暂无数据"

**图表生成**
- 根据数据特征自动选择图表类型
- 使用 ECharts 嵌入交互式图表

**文本生成**
- 核心概述：提炼关键发现
- 分析章节：结合数据和洞察
- 结论建议：基于分析结果

### 6.4 HTML 输出

**基础结构**
- 响应式布局
- 目录导航
- 交互式图表
- 打印优化

**输出目录**
```
output/
├── report.html        # HTML 报告
└── assets/            # 资源文件
    ├── images/
    ├── charts/
    └── styles/
```

## 七、记忆系统设计

### 7.1 三层记忆架构

| 层次 | 生命周期 | 内容 | 隔离策略 |
|------|---------|------|---------|
| 会话记忆 | 单次任务 | 操作历史、已访问 URL | 完全隔离 |
| 模式记忆 | 跨会话 | 网站布局模式、操作技能 | 共享但隔离访问 |
| 知识记忆 | 长期 | 网站特征知识、最佳实践 | 完全共享 |

### 7.2 存储位置

**会话记忆**：存储在任务文件夹的 `memory/` 目录

**模式记忆和知识记忆**：存储在 `shared/` 目录

```
shared/
├── patterns/                   # 模式记忆
│   ├── layouts/                # 布局模式
│   ├── sites/                  # 网站模式
│   └── skills/                 # 操作技能
│
└── knowledge/                  # 知识记忆
    ├── site-features.json      # 网站特征
    └── best-practices.json     # 最佳实践
```

### 7.3 记忆使用

**读取流程**
1. 先检查会话记忆（当前任务）
2. 再检查模式记忆（跨任务共享）
3. 最后检查知识记忆（通用知识）

**写入流程**
1. 会话记忆：实时写入任务文件夹
2. 模式记忆：任务成功完成后写入共享目录
3. 知识记忆：经过验证后写入共享目录

## 八、完成度评估机制

### 8.1 多维度评估

| 维度 | 评估内容 | 达标标准 |
|------|---------|---------|
| 覆盖度 | 子主题信息覆盖情况 | ≥ 80% |
| 深度 | 信息详细程度 | ≥ L3（有数据支撑） |
| 可信度 | 信息来源质量 | 多来源交叉验证 |
| 时效性 | 信息新鲜度 | 符合行业特点 |

### 8.2 终止判断策略

**目标驱动终止**
- 覆盖度、深度、可信度均达标

**边际收益终止**
- 继续采集的边际收益低于阈值
- 连续多次搜索结果重复率高

**资源限制终止**
- 达到时间限制
- 达到页面访问限制

**用户确认终止**
- 不确定时询问用户

## 十、关键技术决策

### 10.1 任务隔离方式

**决策**：基于文件夹隔离

**原因**：
- 简单直观，易于理解和调试
- 完全隔离，任务之间互不干扰
- 易于追溯，保留完整的任务历史
- 支持恢复，可从任意点恢复任务

### 10.2 模板选择方式

**决策**：LLM 智能选择

**原因**：
- 理解语义，判断更准确
- 无需维护复杂规则
- 适应各种边界情况

### 10.3 报告输出格式

**决策**：以 HTML 为主

**原因**：
- 表达能力最强，支持复杂布局和交互
- 用户友好，可直接在浏览器预览
- 生态丰富，有成熟的渲染工具
- 后续可扩展转换为 PDF、PPT

### 10.4 记忆系统架构

**决策**：三层记忆架构

**原因**：
- 会话记忆保证任务隔离
- 模式记忆支持跨任务复用
- 知识记忆支持长期学习

### 10.5 OpenCLI 集成策略

**决策**：作为底层工具集成

**原因**：
- OpenCLI 提供成熟的浏览器命令
- 预置适配器可直接使用
- 支持生成确定性适配器，零 LLM 成本
