# OpenClaw 市场调研智能体系统 - 思考与讨论总结

## 一、整体思路

### 1.1 场景定位

**核心场景**：用户通过自然语言输入（关键词或指定网站），智能体自主进行市场调研，生成结构化报告。

**场景特点**：
- 输入形式多样：关键词调研、指定网站分析、混合输入
- 输出形式丰富：HTML、PDF、PPT、Markdown 等
- 支持迭代交互：追问、修改、格式转换

### 1.2 核心定位变化

**最初定位**：为市场调研场景创建一个独立的 Skill
**最终定位**：将 OpenClaw 改造成适合市场调研-报告生成的智能体系统

**变化原因**：
- Deep Browser 是核心能力，需要深度集成到系统中
- 记忆系统需要跨模块共享
- 规划和判断能力需要与 Agent Engine 紧密配合

### 1.3 设计原则

1. **综合增强**：合理使用所有方案，包括核心改造、Skill 扩展、工具集成等
2. **渐进式改造**：分阶段实现，先验证核心价值再增强功能
3. **学习型系统**：通过记忆机制，让系统越用越聪明
4. **成本优化**：借鉴 OpenCLI 思路，一次学习，多次零成本执行

## 二、讨论过程记录

### 2.1 场景描述讨论

**第一轮**：初步描述使用场景
- 输入形式：关键词调研型、指定网站型、追问迭代型
- 输出形式：摘要 + 链接、WebChat 预览
- 用户反馈：缺少"用户给定网站"的场景

**第二轮**：补充指定网站场景
- 增加了明确指定网站、网站+分析角度、混合输入等场景
- 用户反馈：去掉自动打开浏览器和终端摘要，消息渠道增加飞书

**第三轮**：最终场景描述
- 输入：关键词调研、指定网站、混合输入、追问迭代
- 输出：WebChat 内嵌预览、Gateway 链接访问、文件下载
- 确定了 5 个核心使用场景

**第四轮**：简化输出渠道
- 用户反馈：先不考虑消息渠道的区别
- 统一输出方式：WebChat 预览、Gateway 链接、文件下载

### 2.2 能力需求讨论

**第一轮**：初步能力分析
- 包含了意图理解、信息获取、内容分析、报告生成等能力
- 用户反馈：描述不清晰，包含了通用能力，应聚焦场景专属能力

**第二轮**：重新定义场景专属能力
- 网页理解与操作能力
- 智能规划与调整能力
- 上下文感知与记忆能力
- 报告结构与布局能力
- 多源信息整合能力
- 格式转换与输出能力

**第三轮**：引入 Deep Browser 概念
- 用户提出：需要一个 deep_browser 的能力
- 核心目的：让智能体深度理解/操作/观察效果/获取记忆
- 重新组织能力需求，以 Deep Browser 为核心

### 2.3 方案设计讨论

**第一轮**：Skill 扩展方案
- 结论：不需要修改 OpenClaw 核心，通过 Skill 扩展
- 记忆系统：Skill 内实现文件存储

**第二轮**：引入 OpenCLI 思路
- 用户提出：参考 OpenCLI 的设计
- 核心发现："编译期智能 vs 运行期智能"
- 一次生成适配器，后续执行零 LLM 成本

**第三轮**：网站模式库和操作技能沉淀
- 用 MD 文件描述网站布局模式
- 用 OpenCLI 生成 JS 适配器
- 积累操作经验，越用越聪明

**第四轮**：定位调整
- 用户反馈：不是为场景单独做个 Skill，而是改造 OpenClaw 本身
- Deep Browser 成为系统级能力模块
- 方案调整为综合改造，合理使用所有方案

### 2.4 深度能力讨论

**开放性问题讨论**：
1. 智能体需要吸收哪些记忆？
   - 任务记忆：已完成什么、正在做什么、还需要做什么
   - 信息质量记忆：来源可信度、信息新鲜度、信息完整度
   - 网站特征记忆：结构特征、访问技巧、内容价值
   - 用户偏好记忆：显性偏好、隐性偏好、历史偏好

2. 如何判断工作完成是否到位？
   - 多维度评估：覆盖度、深度、可信度、时效性
   - 终止判断策略：目标驱动、边际收益、资源限制、用户确认
   - 自我评估机制：阶段性自检、报告前终检

3. 数据分析需要扩展哪些能力？
   - 数据提取：结构化数据、数值数据、关系数据
   - 数据验证：交叉验证、逻辑验证、异常检测
   - 数据整合：标准化、聚合、补全
   - 数据可视化：图表类型选择、图表生成

4. 如何优化浏览器使用？
   - 访问策略优化：访问方式选择、批量访问、访问节流
   - 页面理解优化：页面结构分析、内容提取、动态内容处理
   - 浏览器资源管理：内存管理、网络资源管理、会话管理
   - 智能导航策略：链接价值评估、探索深度控制、探索广度控制

## 三、关键记忆点

### 3.1 Deep Browser 核心概念

**定位**：OpenClaw 系统级别的浏览器能力增强模块

**四大核心能力**：
1. **理解能力**：页面类型识别、页面结构理解、交互元素识别、数据结构识别
2. **操作能力**：基础操作、复合操作、策略操作
3. **观察能力**：变化检测、效果验证、状态感知、异常识别
4. **记忆能力**：会话记忆、模式记忆、知识记忆

**工作循环**：目标 → 理解 → 规划 → 操作 → 观察 → 记忆 → 判断完成

### 3.2 OpenCLI 借鉴思路

**核心哲学**：编译期智能 vs 运行期智能

- 传统方案：每次执行都消耗 LLM，成本高、不稳定
- OpenCLI 方案：一次生成适配器，后续执行零 LLM 成本、100% 稳定

**借鉴点**：
- 网站模式库：预定义通用模式 + 学习特定网站模式
- 操作技能沉淀：首次 LLM 决策 → 成功后保存模式 → 后续零 LLM 执行
- 确定性适配器：对于频繁访问的网站，生成适配器

### 3.3 记忆系统设计

**三层架构**：

| 层次 | 生命周期 | 内容 | 用途 |
|------|---------|------|------|
| 会话记忆 | 单次任务 | 操作历史、已访问 URL、当前状态 | 避免重复、支持恢复 |
| 模式记忆 | 跨会话 | 网站布局模式、操作技能模板 | 加速访问、复用经验 |
| 知识记忆 | 长期 | 网站特征知识、最佳实践 | 泛化到新网站 |

### 3.4 完成度评估机制

**多维度评估**：
- 覆盖度：子主题信息覆盖情况
- 深度：信息详细程度（L1-L4）
- 可信度：信息来源质量
- 时效性：信息新鲜度

**终止判断策略**：
- 目标驱动终止：各项指标达标
- 边际收益终止：继续采集收益低于阈值
- 资源限制终止：达到时间或资源限制
- 用户确认终止：不确定时询问用户

## 五、后续工作方向

### 5.1 短期工作

1. **实现 Deep Browser 基础能力**：基础理解、基础操作、会话记忆
2. **实现基础规划能力**：简单关键词规划
3. **实现基础报告生成**：单一模板、基础布局
4. **测试验证**：用实际场景测试验证

### 5.2 中期工作

1. **增强 Deep Browser**：观察能力、复合操作、模式记忆
2. **增强规划能力**：动态调整、完成度评估
3. **增强数据处理**：数据验证、数据整合
4. **OpenCLI 集成**：调用浏览器命令、利用预置适配器

### 5.3 长期工作

1. **智能能力**：自动发现网站数据结构、策略操作、自我评估
2. **知识记忆**：泛化到新网站
3. **OpenCLI 深度集成**：生成确定性适配器
4. **生态整合**：与更多外部工具和服务整合

## 七、核心结论

1. **定位明确**：将 OpenClaw 改造成适合市场调研-报告生成的智能体系统
2. **核心能力**：Deep Browser 提供深度浏览器操作能力
3. **学习机制**：三层记忆系统，让系统越用越聪明
4. **成本优化**：借鉴 OpenCLI 思路，一次学习，多次零成本执行
5. **渐进实现**：分阶段实现，先验证核心价值再增强功能
